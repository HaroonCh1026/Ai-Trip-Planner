# VoyageurAI ML — Cost Prediction Service

This folder contains a Python-based microservice that predicts total trip cost
in PKR for Pakistani travel itineraries. It runs as a separate process from
the main Node.js backend and exposes a small HTTP API the backend calls during
trip generation and admin analytics.

## Architecture

```
┌──────────────┐   HTTP   ┌──────────────┐   HTTP    ┌─────────────────┐
│ React (5173) │ ───────> │ Node (5000)  │ ────────> │ Python (5001)   │
│  Frontend    │          │  Backend     │           │  ML service     │
└──────────────┘          └──────────────┘           └─────────────────┘
                                                              │
                                                              ▼
                                                          model.pkl
                                                          (gradient boost)
```

The model is a gradient-boosted regressor trained on a curated dataset of 5,000
Pakistani trips. On the held-out 20% test set it achieves:

- **R² = 0.96** — explains 96% of cost variation
- **MAE ≈ PKR 27,000** — average error of ~11% on a typical PKR 250k trip
- **RMSE ≈ PKR 44,000** — used as the +/- range when displaying predictions

## One-time setup

### 1. Install Python dependencies

```bash
cd ML
pip install -r requirements.txt
```

If `pip` complains about system-managed packages on newer Linux distros, use:

```bash
pip install -r requirements.txt --break-system-packages
```

Or set up a virtual environment:

```bash
python -m venv venv
source venv/bin/activate    # on Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Train the model

```bash
python train.py
```

This loads `data/pakistan_travel_dataset.csv`, trains three candidate models
(linear regression, random forest, gradient boost), picks the best by R² on
held-out test data, and saves the winner as `model.pkl`. Takes about 30 seconds.

You only need to run this once — and again whenever the dataset is updated.

You should see output like:

```
[1/4] Loading dataset...
  Loaded 5,000 rows, 22 columns
[2/4] Splitting train/test (80/20)...
[3/4] Training candidate models...
  Linear Regression          MAE=PKR     77,175  ...  R²=0.7507
  Random Forest              MAE=PKR     29,944  ...  R²=0.9509
  Gradient Boosting          MAE=PKR     27,092  ...  R²=0.9576
  >>> Winner: Gradient Boosting  (R² = 0.9576)
[4/4] Saving model + metadata...
```

## Running the service

```bash
python service.py
```

Service starts on `http://127.0.0.1:5001`. Leave this terminal open during
development. The Node backend (port 5000) calls it automatically.

## API endpoints

### `GET /health`

Liveness check.

### `GET /meta`

Returns model metadata — performance metrics, supported categories,
training date.

### `POST /predict`

Predict total cost for a single trip. Body:

```json
{
  "distance_km": 605,
  "duration_days": 7,
  "traveler_age": 30,
  "group_size": 4,
  "region": "Gilgit-Baltistan",
  "traveler_gender": "Male",
  "traveler_nationality": "Pakistani",
  "trip_type": "Family",
  "accommodation_type": "Mid",
  "transportation_type": "Road (Hiace)",
  "season": "Summer"
}
```

Response:

```json
{
  "success": true,
  "prediction": {
    "predicted_cost_pkr": 300316,
    "low_pkr": 256360,
    "high_pkr": 344271,
    "rmse_pkr": 43955,
    "currency": "PKR"
  }
}
```

### `POST /predict/batch`

Same as `/predict` but accepts up to 1000 trips in one call. Used by the
admin analytics tab to back-fill predictions for historical trips.

## Local dev workflow (3 terminals)

```bash
# Terminal 1 — Node backend
cd Backend && npm run dev

# Terminal 2 — React frontend
cd Front-end && npm run dev

# Terminal 3 — ML service
cd ML && python service.py
```

## Limitations and future work

- **No personalized predictions** — the model predicts based on aggregate
  patterns from 5,000 trips. It doesn't know the specific user's history.
- **Cost components not predicted individually** — only the total. We could
  train separate sub-models for accommodation/transport/food/activities if
  needed.
- **No live re-training** — model is static after `train.py` runs. To pick up
  new data you must re-run training. A scheduled re-train (weekly cron) is
  on the roadmap.
- **Categorical features only know what they've seen** — if you pass a region
  or trip type the model has never seen, OneHotEncoder treats it as all-zeros.
  This is graceful degradation rather than an error.
